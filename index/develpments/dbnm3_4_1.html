DBNM 3.4.1: Scoring Pipeline Rework and Network Stability

June 2025 Read the full changelog

DBNM 3.4.1 is a focused maintenance release addressing a set of long-standing issues in the scoring pipeline, improving the accuracy of node weight calculations and resolving an edge case that caused incorrect propagation during high-traffic intervals. This document details the changes, the reasoning behind each decision, and what to expect during the transition.

What Changed

The scoring pipeline previously relied on a two-pass evaluation model. In the first pass, raw connection counts were tallied per node. In the second pass, those counts were weighted against a static decay table. This approach worked well in low-volume simulations but introduced compounding errors as node counts scaled past roughly 1,400 active entries.

In 3.4.1, the static decay table has been replaced with a dynamic weight resolver. Instead of referencing a fixed lookup, the resolver re-evaluates decay factors at each interval using a rolling window over the last 40 cycles. This eliminates the drift that accumulated in longer sessions and brings per-node scoring error from ~4.2% down to under 0.8% in internal testing.

Network Stability Improvements

Three separate stability bugs have been resolved in this release. The most impactful involved a race condition in the connection handshake logic that could cause duplicate session records when two nodes attempted to register simultaneously. The fix involved adding a short mutex lock at the registration step — a simple change, but one that required careful sequencing to avoid introducing latency.

The second fix addressed an off-by-one error in the adjacency list indexing that surfaced only when the network was initialized with an even number of seed nodes. The third resolved an issue where the health-check daemon would silently skip nodes flagged as inactive but still present in the routing table, causing stale entries to persist.

Migration Notes

Most users will not need to take any action. If you are running a custom scoring configuration exported from a version earlier than 3.2.0, review the release notes for that version and ensure your configuration file uses the updated resolver_mode key. The old decay_table_path key is still accepted but will log a deprecation warning and will be removed in 3.6.

Sessions in progress at the time of upgrade will complete using the prior scoring model. The new resolver activates on the next full session initialization.

Frequently Asked Questions

Questions collected from issue reports and internal review during the 3.4.1 testing cycle.

Why wasn't the dynamic resolver introduced earlier?

The static decay table was sufficient for the scale DBNM was originally designed for. As the test network grew beyond a few hundred nodes, the drift became measurable but was not a blocking issue. The rework became a priority once error rates started affecting leaderboard accuracy in competitive sessions.

Will this change scoring results for existing data?

Historical session records are not retroactively recalculated. Existing scores remain as-is. Only new sessions initiated after the upgrade will use the dynamic resolver.

What does the rolling window size of 40 cycles mean in practice?

The resolver looks at the last 40 evaluation cycles when computing each decay factor. At the default tick rate this is approximately 8 seconds of history. A smaller window would make scores more reactive but less stable; a larger window smooths out spikes but can lag behind rapid topology changes. 40 was chosen as a balance point based on observed network behavior.

Is there a performance cost to the new resolver?

Yes, a small one. Benchmarks show roughly a 3–5% increase in CPU time during the scoring phase on large networks. This is within acceptable limits and is far outweighed by the accuracy gains. A future release may introduce an optional fast-path for networks below a configurable node threshold.

Where can I report issues found in 3.4.1?

Open an issue through the standard reporting channel. Include your configuration file (with sensitive fields redacted), the session log from the affected run, and the exact version string from the about page.
